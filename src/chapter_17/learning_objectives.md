## Learning Objectives

By the end of this chapter, you will be able to:

1. Explain why Software 3.0 systems require a distinct architectural approach from traditional software and from Software 2.0 machine learning systems.
2. Describe the four layers of the LLM application stack — prompt, retrieval, orchestration, and guardrail — and explain the engineering responsibility of each.
3. Analyse how context window constraints shape fundamental architecture decisions, including the lost-in-the-middle problem.
4. Reason about token economics and calculate the cost implications of architectural choices.
5. Distinguish LLM failure modes — hallucination, refusal, sycophancy, and context drift — from traditional ML failure modes and identify the engineering response to each.
6. Navigate the three-way tradeoff between latency, cost, and quality when designing LLM systems.
7. Design stateful LLM applications with explicit session management, conversation history, and memory strategies.
8. Apply these architectural principles to MedChat, the clinical chatbot case study used throughout Chapters 17–21.
9. Articulate why API-first design on foundation models shifts engineering responsibility from model behaviour to application behaviour.
