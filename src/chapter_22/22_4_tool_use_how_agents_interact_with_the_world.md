## 22.4 Tool Use: How Agents Interact with the World

---

Tools are the mechanism by which an agent extends its capabilities beyond what is encoded in the language model's weights. Without tools, an agent can reason and plan but cannot perceive the current state of the world or affect it. Tools give the agent eyes, hands, and a calendar.

A tool, in technical terms, is a function with a defined schema. The schema specifies the tool's name, a natural-language description of what it does, and the parameters it accepts with their types and constraints. The language model uses the schema — not the implementation — to decide when and how to call the tool. This means tool schema design is a first-class engineering concern: a poorly described tool will be misused, a poorly constrained schema will produce invalid calls, and an ambiguously named tool will be selected at the wrong time.

Three principles govern good tool schema design. First, tool names and descriptions should be precise enough to be unambiguous but short enough to fit comfortably in a prompt that may include many other tool definitions. A tool called `search` is dangerously underspecified in a system with multiple retrieval tools. A tool called `search_patient_medication_history_by_id_and_date_range` is more precise but cognitively heavy. The right balance is descriptive specificity without verbosity: `get_medication_history` with parameters `patient_id` and `date_range` communicates both function and interface clearly. Second, parameter constraints should be explicit — required fields, allowed value ranges, enumerated options — because the model will attempt to fill any unconstrained parameter with its best guess, which may be invalid. Third, the tool's failure mode should be documented in the schema: what does the tool return when it finds nothing, when the service is unavailable, or when the parameters are out of range? An agent that does not know how a tool signals failure cannot interpret a failure response correctly.

### 22.4.1 The Read/Write Distinction

The single most important taxonomic distinction in tool design is between read tools and write tools. This distinction determines the risk profile of every tool call and drives the safety architecture of the entire system.

**Read tools** retrieve information from the world without modifying it. A database query, a document retrieval, a web search, a medical literature lookup — these are all read tools. Their defining property is that repeated calls with the same parameters produce the same result (or results that change only because the underlying data has changed independently). Read tools can be called freely during the Reason and Act phases because their failure modes are bounded: the worst outcome of an incorrect read-tool call is that the agent receives irrelevant or absent information, which it must then reason about. A read-tool error is recoverable: the agent can notice the inadequate result and call a different tool or reformulate the query.

**Write tools** modify the state of the world: inserting a record into a database, sending a notification to a patient, prescribing a medication, executing a financial transaction, modifying a file, or calling an external API that triggers a downstream process. Write tools are not freely callable. They require validation before execution, may be partially or fully irreversible, and their failure modes extend beyond the agent system into the external world.

Three engineering controls apply specifically to write tools. First, **idempotency**: wherever possible, write tools should be designed so that calling them multiple times with the same parameters produces the same result as calling them once. If an agent retries a failed write operation, it should not create duplicate records or trigger duplicate external events. Idempotency is not always achievable — sending an email twice genuinely sends two emails — but it should be designed into every write tool where the domain permits. Second, **validation gates**: write tools should validate their parameters before acting, reject invalid inputs with specific error messages (not generic failures), and — for consequential actions — require explicit confirmation from a human or a separate validation agent before executing. Third, **audit logging**: every write-tool call should produce an immutable log entry recording the parameters, the caller identity, the timestamp, and the outcome. This is not optional for production systems. It is the mechanism by which an agent's actions become accountable.

### 22.4.2 Safety Constraints on Tool Design

Tool safety is not solely about whether individual tools are safe — it is about whether the composition of tools is safe. An agent that can read patient records and can send emails has, in combination, the ability to leak private health information outside the system. Neither tool is unsafe in isolation; their combination creates a risk that must be controlled at the system level.

The recommended approach is a tool permission model that specifies which combinations of tools are permitted for which agent roles. A clinical summary agent may have read access to medication history and write access to summary reports, but should not have access to communication tools that could route information outside the system. Tool permissions are a form of capability restriction analogous to role-based access control in traditional software systems, and they should be defined before any agent is deployed, not retrofitted after an incident.
